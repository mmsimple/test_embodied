# 机器人测试工程师学习路线

## 目标定位

作为机器人测试工程师，核心目标不是一开始成为算法专家，而是尽快具备以下能力：

- 看懂机器人系统的基本组成和运行流程
- 能设计合理、可复现、可量化的测试场景
- 能定位问题发生在哪一层：硬件、驱动、感知、规划、控制、通信或业务逻辑
- 能通过日志、数据包、可视化工具和自动化脚本分析问题
- 能推动问题从发现、复现、定位到关闭

机器人测试相比普通软件测试，更强调：

- 可复现
- 可量化
- 可回放
- 可追踪
- 安全性
- 长时间稳定性

## 一、机器人基础

### 1. 机器人系统组成

需要了解机器人通常由哪些部分构成：

- 传感器：相机、深度相机、激光雷达、IMU、编码器、力传感器等
- 执行器：电机、舵机、机械臂、夹爪、灵巧手、移动底盘等
- 控制器：工控机、嵌入式控制板、PLC、运动控制器等
- 通信系统：CAN、EtherCAT、串口、以太网、Wi-Fi 等
- 软件系统：感知、定位、建图、规划、控制、任务调度、人机交互等

### 2. 坐标系与变换

机器人系统中坐标系非常重要，测试时经常要判断定位、感知和动作是否对齐。

重点学习：

- 世界坐标系
- 机器人本体坐标系
- 相机坐标系
- 雷达坐标系
- 工具坐标系
- 末端执行器坐标系
- 坐标变换
- TF 坐标树

需要能理解：

- 一个物体在相机中看到的位置，如何转换到机器人坐标系
- 机器人移动后，地图、机器人、传感器之间的关系是否正确
- 坐标系错位可能导致导航偏移、抓取失败、避障异常等问题

### 3. 运动学和控制基础

不需要一开始深入推公式，但要理解基本概念。

重点学习：

- 关节
- 自由度
- 机械臂正运动学
- 机械臂逆运动学
- 轨迹
- 速度
- 加速度
- PID 控制
- 闭环控制
- 位置控制
- 速度控制
- 力控

测试时需要关注：

- 机械臂是否能到达目标点
- 轨迹是否平滑
- 是否出现抖动、超调、震荡
- 是否触发关节限位
- 底盘是否打滑、漂移、偏航

### 4. 传感器基础

常见传感器及测试关注点：

| 传感器 | 关注点 |
| --- | --- |
| RGB 相机 | 图像清晰度、曝光、延迟、丢帧 |
| 深度相机 | 深度误差、空洞、反光物体表现 |
| 激光雷达 | 点云质量、遮挡、噪声、时间戳 |
| IMU | 姿态漂移、噪声、频率稳定性 |
| 编码器 | 里程计误差、速度反馈、丢脉冲 |
| 力传感器 | 零漂、灵敏度、过载保护 |

### 5. 标定与时间同步

机器人测试中，很多问题都和标定或同步有关。

重点学习：

- 相机内参
- 相机外参
- 相机标定
- 手眼标定
- 传感器外参
- 时间戳
- 多传感器时间同步

常见问题：

- 感知结果位置不准
- 抓取点偏移
- 雷达点云和地图错位
- 机器人运动时图像、点云和里程计不同步

## 二、ROS / ROS2

ROS 或 ROS2 是机器人开发和测试中非常常见的工具链。

### 1. 核心概念

需要掌握：

- node
- topic
- service
- action
- message
- launch
- parameter
- bag
- tf / tf2

### 2. ROS2 常用命令

```bash
ros2 node list
ros2 node info /node_name
ros2 topic list
ros2 topic echo /topic_name
ros2 topic hz /topic_name
ros2 topic info /topic_name
ros2 service list
ros2 action list
ros2 param list
ros2 bag record /topic_name
ros2 bag play bag_file
```

### 3. ROS1 常用命令

如果公司使用 ROS1，需要掌握：

```bash
rosnode list
rosnode info /node_name
rostopic list
rostopic echo /topic_name
rostopic hz /topic_name
rosservice list
rosparam list
rosbag record /topic_name
rosbag play bag_file
roslaunch package_name launch_file.launch
```

### 4. 必学工具

- RViz：查看机器人模型、TF、地图、点云、轨迹、目标点
- Gazebo：仿真测试
- Isaac Sim：高级仿真和具身智能场景
- rosbag / ros2 bag：录制和回放数据
- rqt_graph：查看节点关系
- rqt_plot：绘制话题数据曲线

### 5. ROS 测试重点

需要能做到：

- 看当前有哪些节点在运行
- 判断 topic 是否正常发布
- 判断发布频率是否稳定
- 录制问题现场 bag
- 回放 bag 复现问题
- 在 RViz 中查看 TF 是否正确
- 判断感知、规划、控制模块的数据链路是否正常

## 三、测试工程能力

### 1. 测试用例设计

需要掌握常见测试类型：

- 功能测试
- 边界测试
- 异常测试
- 回归测试
- 稳定性测试
- 压力测试
- 安全测试
- 兼容性测试
- 性能测试

机器人测试用例应包含：

- 测试目标
- 测试环境
- 软件版本
- 硬件配置
- 初始状态
- 操作步骤
- 期望结果
- 实际结果
- 判定标准
- 日志和数据附件

### 2. Bug 描述规范

一个高质量 Bug 应包含：

- 问题标题
- 问题现象
- 期望结果
- 实际结果
- 复现步骤
- 复现概率
- 影响范围
- 发生环境
- 软件版本
- 硬件版本
- 日志
- 截图或视频
- rosbag 或其他数据包

示例结构：

```markdown
## 问题标题

机器人在窄通道避障时停止后无法恢复导航

## 环境

- 机器人型号：
- 软件版本：
- 地图版本：
- 测试场地：
- 传感器配置：

## 复现步骤

1. 启动导航任务
2. 在机器人前方 0.5 m 放置障碍物
3. 等待机器人停止
4. 移除障碍物
5. 观察机器人是否恢复运动

## 期望结果

障碍物移除后，机器人应在 3 秒内重新规划路径并继续执行任务。

## 实际结果

机器人保持停止状态，未继续导航。

## 复现概率

8/10

## 附件

- 日志：
- 视频：
- rosbag：
```

### 3. 稳定性测试

机器人稳定性测试通常包括：

- 连续运行 8 小时、24 小时或更长时间
- 多轮任务循环
- 长时间导航
- 重复抓取
- 高频任务下发
- 网络波动
- 电量变化
- 温度变化

关注指标：

- 崩溃次数
- 任务成功率
- 平均无故障时间
- CPU 占用
- 内存占用
- 传感器频率
- 通信延迟
- 电池电压
- 电机温度

### 4. 安全测试

机器人测试必须重视安全。

常见安全测试项：

- 急停按钮是否有效
- 软件急停是否有效
- 碰撞检测是否有效
- 障碍物避让是否有效
- 关节限位是否有效
- 速度限制是否生效
- 断网后是否进入安全状态
- 传感器异常后是否停止或降级
- 电量过低时是否安全返回或停止

## 五、常见机器人测试方向

### 1. 移动机器人导航测试

测试内容：

- 定位是否稳定
- 建图是否准确
- 路径规划是否合理
- 动态避障是否有效
- 窄通道是否能通过
- 是否会原地打转
- 是否能从异常中恢复

核心指标：

- 到达率
- 平均耗时
- 路径长度
- 停滞次数
- 碰撞次数
- 定位丢失次数

### 2. 机械臂测试

测试内容：

- 点位精度
- 重复定位精度
- 轨迹平滑性
- 速度和加速度限制
- 关节限位
- 碰撞保护
- 末端工具控制

核心指标：

- 到位误差
- 重复误差
- 任务成功率
- 抖动幅度
- 超限次数

### 3. 抓取测试

测试内容：

- 目标识别
- 位姿估计
- 抓取点生成
- 机械臂到位
- 夹爪闭合
- 抬起是否掉落
- 失败恢复策略

核心指标：

- 识别准确率
- 抓取成功率
- 掉落率
- 平均抓取耗时
- 失败原因分布

### 4. 感知测试

测试内容：

- 目标检测
- 图像分割
- 深度估计
- 点云处理
- 姿态估计
- 多传感器融合

核心指标：

- 漏检率
- 误检率
- 检测精度
- 深度误差
- 帧率
- 延迟

### 5. 具身智能和大模型测试

如果公司涉及具身智能或机器人智能体，需要关注：

- 语言指令理解是否准确
- 任务拆解是否合理
- 规划步骤是否可执行
- 是否会产生不安全动作
- 是否能处理失败反馈
- 是否能在真实环境中完成闭环任务

示例测试任务：

- “把桌上的杯子拿给我”
- “找到红色积木并放到盒子里”
- “去厨房门口等待”
- “整理桌面上的物品”

测试重点：

- 指令理解准确率
- 任务完成率
- 失败恢复能力
- 安全边界
- 泛化能力
- 响应延迟

- 测试用例设计

产出：

- 能设计完整测试方案

产出：

## 总结

机器人测试工程师的核心竞争力不是只会点按钮执行测试，而是能把复杂的机器人系统拆解成可测试、可复现、可量化的问题。

短期目标是熟悉工具和流程，能快速上手测试任务。

中期目标是能独立负责模块测试，并用脚本和数据提高效率。

长期目标是建立系统化的机器人质量保障能力，成为连接研发、产品、现场和用户问题之间的关键角色。
